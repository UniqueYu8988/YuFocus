# AGENTS.md

本文件是 AI 进入项目后的第一操作入口和导航页。
它规定“怎样工作”，不重复保存完整产品、架构和进度信息。

## 1. 用户背景

项目负责人不是计算机专业人员。

- 使用清晰、具体的中文；
- 专业术语首次出现时解释；
- 优先选择简单、稳定、容易维护的方案；
- 重大技术选择说明原因、代价和未来影响；
- 不要把“更复杂”当作“更专业”。

## 2. 项目事实来源

开始工作前，根据任务需要阅读：

- 产品行为和范围：`PRODUCT.md`
- 技术结构和数据流：`ARCHITECTURE.md`
- 当前状态和下一步：`CURRENT_STATE.md`
- 当前行为基线：`docs/BASELINE_ACCEPTANCE.md`
- 稳定化阶段：`docs/plans/STABILIZATION_PLAN.md`
- 当前任务：`docs/tasks/`
- 重要技术决定：`DECISIONS.md`（如果存在）
- 对话交接：`HANDOFF.md`（如果存在）

代码和测试是最终真实状态。文档与代码冲突时，先报告并核对，不要擅自选择其中一方。

### 日常入口优先级

新对话或新任务优先按下面顺序进入：

1. `AGENTS.md`：确认工作规则、任务规模和安全红线；
2. `CURRENT_STATE.md`：确认当前阶段、下一步和暂时不做的内容；
3. `PRODUCT.md`、`ARCHITECTURE.md`：只在任务涉及产品行为或技术结构时读取；
4. `docs/BASELINE_ACCEPTANCE.md`：只在需要验证行为、启动、数据安全或回归风险时读取；
5. `docs/plans/STABILIZATION_PLAN.md`：只在任务影响稳定化阶段或优先级时读取；
6. `docs/tasks/`：中型和大型任务的具体入口。

`docs/VERIFICATION_BACKLOG.md`、`docs/history/legacy-md/` 中的旧资料、`README.md` 和历史报告只能作为参考线索，不是当前事实的最高来源。引用其中内容前，必须通过当前代码、当前文档或项目负责人确认。

## 3. 每次任务开始前

1. 确认项目根目录；
2. 阅读与任务相关的项目文档；
3. 检查目录结构；
4. 如果项目已经使用 Git，检查当前状态和最近提交；
5. 找出相关代码、数据和测试；
6. 先复述目标、范围、明确不做的内容和风险；
7. 未理解当前行为前，不进行大规模修改。

## 4. 任务规模

### 小型任务

局部、影响明确、不改变架构和数据、可以一次完成和验证。可直接在对话中定义，不强制创建任务文件。

### 中型任务

符合任一项时，在 `docs/tasks/` 创建任务文件：

- 涉及多个文件；
- 改变用户可观察行为；
- 需要增加测试；
- 可能分多次完成；
- 可能切换对话；
- 会影响现有流程。

### 大型任务

中型任务基础上，符合任一项时还要创建计划：

- 修改数据库结构；
- 登录、权限、支付或隐私；
- 核心大文件拆分；
- 引入重要依赖；
- 更换框架或基础设施；
- 涉及真实数据迁移；
- 无法通过一个安全小步骤完成。

大型任务需要：

- `docs/tasks/具体任务.md`
- `docs/plans/具体计划.md`

## 4.1 固定工作流程

每个普通任务按这个顺序执行：

1. 开始：读 `AGENTS.md` 和 `CURRENT_STATE.md`，再按任务需要读取产品、架构、验收或任务文件。
2. 定界：用简短中文复述目标、允许做什么、明确不做什么、主要风险。
3. 实施：小步修改，只碰本任务需要的文件；遇到安全红线先停下报告。
4. 验证：运行与改动相关的最小测试、检查或人工验收；不能运行时说明原因。
5. 收尾：更新必要 Markdown，说明改了什么、验证结果、风险、下一步。

如果只是回答问题或只做审计，可以不修改文件；如果进入中型或大型任务，先建立任务文件或计划，不直接开改。

## 5. 修改原则

- 一次只解决一个主要问题；
- 优先最小、可验证、可回退的改动；
- 不顺便修改无关代码；
- 不同时进行新功能开发和大规模重构；
- 不未经说明新增依赖；
- 不隐藏测试失败；
- 不把新功能继续堆入职责混乱的大文件；
- 重构时保持用户可观察行为不变；
- 删除代码前确认没有调用者和数据风险。

## 6. 老项目稳定化规则

稳定化前先建立：

1. 事实基线：真实功能和已知问题；
2. 运行基线：怎样启动、当前错误和环境要求；
3. 数据基线：数据库、上传文件、配置和备份位置；
4. 验收基线：核心流程如何人工或自动验证。

未建立基本验收方式前，不进行核心逻辑大重构。

## 7. 验证规则

完成修改前，按项目实际能力执行：

- 相关自动测试；
- 类型检查；
- 构建；
- 静态检查；
- 与改动有关的人工验收；
- Git diff 检查（如果已使用 Git）。

不能只说“应该可以”。未能执行的验证必须明确说明。

## 8. 文档更新原则

文档按事实变化更新，不机械更新全部文件。

- 用户行为、范围变化：更新 `PRODUCT.md` 或 `CHANGELOG.md`；
- 模块、目录、依赖、数据流变化：更新 `ARCHITECTURE.md`；
- 当前任务、问题或下一步变化：更新 `CURRENT_STATE.md`；
- 重要且难以撤销的技术选择：更新 `DECISIONS.md`；
- 切换对话或暂停项目：更新 `HANDOFF.md`；
- 获得值得长期保留的知识：更新 `LEARNING_LOG.md`；
- 阶段目标变化：更新 `ROADMAP.md` 或稳定化计划；
- 如果对应事实没有变化，不要为了流程而修改文件。

### 文档职责边界

- `PRODUCT.md` 只写产品行为、边界和用户可见能力；
- `ARCHITECTURE.md` 只写模块、数据流、启动方式、依赖和数据位置；
- `CURRENT_STATE.md` 只写当前阶段、当前风险、下一步和暂停事项；
- `docs/BASELINE_ACCEPTANCE.md` 只写核心验收，不写产品设想和历史长清单；
- `docs/plans/STABILIZATION_PLAN.md` 只写稳定化阶段，不写具体代码重构方案；
- `docs/VERIFICATION_BACKLOG.md` 只放待验证候选问题，不等同于当前 Bug；
- `docs/history/legacy-md/` 只作历史线索，日常不要作为入口；
- `docs/history/legacy-md/PROJECT_CONTEXT.md` 和 `README.md` 可能包含旧路线，使用前必须和当前基线文档核对。

## 9. 安全红线

遇到下列情况先停止并报告：

- 可能删除或覆盖真实数据；
- 可能提交密码、密钥、令牌或证书；
- 项目入口、数据位置或当前行为不明确；
- 修改范围明显超出任务；
- 测试失败且原因不清楚；
- 需要执行不可逆命令；
- 需要整文件重写核心代码；
- 发现文档与代码严重冲突。

## 10. 完成报告

按任务规模提供：

- 完成了什么；
- 修改了哪些文件；
- 用户可观察变化；
- 验证结果；
- 风险和遗留问题；
- 如何回退；
- 本次涉及的一个技术概念。
