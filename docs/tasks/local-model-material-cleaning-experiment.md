# 本地模型资料层清洗实验

创建日期：2026-06-30

## 背景

当前正式链路中，MiMo 负责 `exports/notebooklm.md` 主资料清洗，本地 Ollama 模型负责 `brief.local.md` / 邮件等消费层输出。

现在需要验证一个关键问题：如果只是把音频转译后的原始字幕整理成“完整、可追溯、适合 NotebookLM 导入”的资料稿，本地模型是否可以承担这一层，或者至少承担前置清洗层。

## 目标

用同一条马督工视频的原始字幕，分别生成：

1. 现有 MiMo / 正式清洗对照稿；
2. 本地模型单次长上下文清洗稿；
3. 本地模型分块清洗 + 合并稿。

然后比较它们在以下方面的差距：

- 是否保留主要信息；
- 是否保留时间顺序和时间戳；
- 是否出现“略 / 省略 / 总结式压缩”；
- 是否改错事实、数字、人名、机构名；
- 是否输出截断；
- 处理耗时；
- 本机承压情况；
- 是否适合接入正式主流程。

## 样本

优先沿用上一轮马督工实验样本：

- UP：马督工
- BVID：`BV15CTg6uE4U`
- 标题：`【睡前消息1071】电动汽车交养路费 别讲“4次方”鬼故事`
- 正式材料目录：`data/materials/316568752/bv15ctg6ue4u`

## 实验边界

- 只写入 `data/temp/local-material-cleaning-experiment/`。
- 不覆盖正式 `data/materials`。
- 不修改主流程代码。
- 不自动加入队列。
- 不发送邮件。
- 不打印 API Key、Cookie、SMTP 授权码等秘密值。

## 初始实验设计

### A. 对照稿

读取现有正式 `exports/notebooklm.md` 作为 MiMo / 正式链路对照。

### B. 本地单次长上下文

把完整原始字幕一次性交给本地模型，逐步提高上下文窗口和输出预算：

- 基线：`num_ctx=4096`；
- 提升：`num_ctx=8192`；
- 如机器可承受，再尝试：`num_ctx=12288` 或更高。

每次记录：

- 是否成功；
- 耗时；
- 输出字数；
- 是否截断；
- 是否出现省略；
- 是否保留关键实体和两个主题。

### C. 本地分块清洗 + 合并

把原始字幕按时间或字符分成多个块，每块要求“只清洗、不总结、不省略”，然后把分块结果合并。

分块方案优先使用：

- 每块约 2500-3500 中文字符；
- 前后保留少量上下文；
- 每块输出保持时间戳和顺序；
- 最终合并阶段只做标题和重复边界清理，不允许二次总结。

### D. 质量判定

初步用自动指标筛查：

- 输出长度比例；
- 关键术语是否缺失；
- 是否出现“略 / 省略 / 总结 / 以下为广告内容”等危险词；
- 是否保留时间戳；
- 是否明显截断。

最后再人工阅读开头、中段、结尾和主题切换处，形成结论。

## 预期决策

实验结束后只做判断，不立即接入主流程：

- 本地模型是否可以直接生成 `notebooklm.md`；
- 或者只适合做 MiMo 前置清洗；
- 或者仍应保持 MiMo 作为资料层主力，本地模型只做消费层。

### 实际测试内容

使用同一条马督工视频的 MiMo 清洗前输入：

`data/materials/316568752/bv15ctg6ue4u/work/cleaning/chunks/clean_chunk_001.input.md`

该输入既包含原始音频转写错误，也包含 `[00:00-00:52]` 这类时间段，因此比 `raw_transcript.txt` 更适合测试资料层清洗。

本次共测试：

1. 本地单次 `num_ctx=4096`、`num_predict=4096`；
2. 本地单次 `num_ctx=8192`、`num_predict=8192`；
3. 本地单次 `num_ctx=12288`、`num_predict=8192`；
4. 本地分块 `num_ctx=8192`、`num_predict=4096`；
5. 术语表辅助的本地分块 `num_ctx=8192`、`num_predict=4096`。

### 主要结果

| 版本 | 耗时 | 字数 | 时间戳数量 | 判断 |
|---|---:|---:|---:|---|
| 正式 MiMo / NotebookLM 对照稿 | 已存在 | 8177 | 32 | 最好 |
| 本地单次 `ctx=4096` | 13.8 秒 | 329 | 0 | 失败 |
| 本地单次 `ctx=8192` | 207.0 秒 | 4111 | 21 | 不完整 |
| 本地单次 `ctx=12288` | 545.3 秒 | 7498 | 32 | 能跑完整，但清洗弱 |
| 本地分块 `ctx=8192` | 395.4 秒 | 7546 | 32 | 长度稳定，质量弱 |
| 术语表辅助分块 `ctx=8192` | 350.5 秒 | 7496 | 32 | 小幅改善，仍弱于 MiMo |

### 结论

本地模型可以通过提高上下文窗口或分块方案处理完整字幕，但当前 `shijie-qwen3-8b-q4-chat` 不适合直接替代 MiMo 生成 `notebooklm.md`。

核心原因不是电脑跑不动，而是本地 8B 模型对音频转写错误的主动修复能力不足。它能保住时间戳和长度，却仍然残留大量 ASR 噪音，例如 `2268星`、`煤大高速`、`未来创始人`、`EO李斌`、`10071期` 等。

当前建议保持：

```text
原始字幕 / 音频转写
  ↓
规则预处理
  ↓
MiMo 高保真清洗：notebooklm.md
  ↓
本地模型消费层：brief.local.md / 邮件摘要 / 价值判断
```

后续如果继续探索本地资料层，应优先测试更强本地模型、UP 词表、自动质量门控和失败回退，而不是直接把当前 8B 模型接入正式资料层。
